# 集群管理

## 集群节点角色

XuguDB 集群中按功能职责划分为以下角色，通过 cluster.ini 的 `ROLE` 参数配置：

### 主控管理角色（M）

- **职责**：集群管理、心跳检测、全局锁仲裁（检测死锁并解锁）
- **数量要求**：必须配置 2 个，两个一组构成主备关系（MASTER_GRPS 目前仅支持 1 组）
- **部署规则**：必须分别配置在奇偶节点，建议保持默认配置
- **特点**：不参与用户请求响应，采用双活冗余设计，一个故障时另一个自动无缝接管
- **与 Oracle RAC 对比**：Oracle RAC 无独立管理节点，由 Cluster Ready Services (CRS) 提供集群管理功能
- **与 MySQL Cluster 对比**：MySQL NDB 的管理节点（ndb_mgmd）类似但通常仅 1 个，XuguDB 强制 2 个以保证高可用

### 工作计算角色（W）

- **职责**：并发并行任务调度与计算处理，生成执行计划，与存储角色交互完成数据读写
- **部署规则**：**W 必须与 Q 联合部署（QW），不存在单独的 W 或单独的 Q**。W 负责任务调度和计算，但不建立 TCP 监听端口，需依赖 Q 角色接入外部请求
- **特点**：所有 QW 节点对等处理，用户连接任一 QW 节点均可访问其权限内的所有数据
- **扩展**：负载过高时可通过增加 QW 节点分担压力
- **与 Oracle RAC 对比**：Oracle RAC 所有实例均为计算节点；XuguDB 将计算角色独立，更灵活
- **与 PostgreSQL Citus 对比**：Citus 的协调节点类似工作角色，但 Citus 协调节点通常仅 1 个（或通过 HA 方案冗余），XuguDB 支持多个工作节点同时服务

### 存储管理角色（S）

- **职责**：存储数据库数据，实现数据持久化、存储高可用、数据快速定位
- **存储机制**：
  - 不以逻辑表和业务维度进行数据切割
  - 以 **8MB** 物理 CHUNK 方式自动切片
  - 主副版本通过内部算法打散均衡分布到各存储节点
- **副本控制**：
  - `default_copy_num`：存储的默认版本个数（分布式版建议 3）
  - `safely_copy_num`：存储的安全版本个数（分布式版建议 2）
- **存储权重**：通过 cluster.ini 的 `STORE_WEIGHT` 参数配置，按磁盘容量比设置
- **与 Oracle RAC 对比**：Oracle RAC 依赖共享存储（ASM），XuguDB 每个存储节点独立存储（Shared-Nothing）
- **与 MySQL NDB 对比**：NDB 数据节点类似但使用内存优先存储，XuguDB 为磁盘持久化存储

### 查询角色（Q）

- **职责**：建立 TCP 监听端口，作为对外服务入口；只有含 Q 角色的节点才能接受外部客户端连接，不含 Q 的节点仅参与集群内部通信
- **适用场景**：Q 是客户端连接的必要条件，所有需要对外服务的节点必须配置 Q 角色
- **部署规则**：**Q 必须与 W 联合部署（QW），不存在单独的 Q 或单独的 W**。Q 提供 TCP 监听和请求接入，W 提供任务调度和计算处理，两者配合完成完整的请求处理流程
- **与 PostgreSQL 对比**：类似 PostgreSQL 的监听服务层，但 XuguDB 将监听接入与计算处理分为独立角色，在集群内原生支持

### 变更收集角色（G）

- **职责**：收集集群数据变更信息，记载信息粒度达到事务级
- **数量限制**：最多允许配置于 2 个节点
- **特点**：
  - 可按需部署（可选）
  - 用户可通过系统内置包解析变更信息
  - 解析后可提供字段数据输出或 SQL 输出
  - 可用于同构或异构表的数据恢复与同步
  - 支持按事务提交时间进行日志记录和时间回溯
- **与 Oracle 对比**：类似 Oracle LogMiner / GoldenGate 的变更捕获功能，但 XuguDB 原生集成在集群中
- **与 MySQL 对比**：类似 MySQL Binlog 的功能定位，XuguDB 的 Binlog 也可用于数据复制和主从同步

## 存算融合部署

XuguDB 支持在一台物理服务器上同时部署多个角色，让一个节点执行多种角色能力：

- 保证所有节点都可以提供读写服务
- 适用于资源有限但需要集群能力的场景
- 典型配置：一个节点同时承担 M + W + S 角色

## 节点管理

### 节点标识

每个节点通过 cluster.ini 中的以下参数标识：

| 参数 | 说明 |
|------|------|
| `NID` | 集群节点号，连续递增 |
| `MY_NID` | 当前节点号，必须与本机 IP 对应的 PORTS 匹配 |
| `PORTS` | 格式 `IP:PORT`，指定节点间通信地址。发送端口 = 接收端口 + 20 |
| `ROLE` | 节点角色组合 |
| `LPU` | 参与运算的最大逻辑 CPU 数（建议物理核数 - 1） |

### 节点启停

```bash
# 启动节点
cd /path/to/XuguDB/Server/BIN
./startdb.sh

# 启动方式选择（编辑 startdb.sh）
# --child    前端监控方式
# --service  后台服务方式
# --server   后台服务方式
```

验证启动状态：查看 `stdout.txt` 日志，监听成功即服务正常。

### 节点降级

通过 xugu.ini 参数 `enable_node_degrade = true` 控制是否启用节点降级。当节点发生部分故障但未完全宕机时，可自动降级以维持集群整体可用性。

## 在线扩缩容

### 扩容（添加节点）

XuguDB 分布式版支持在线动态扩展：

1. 在新服务器上创建 xugu 用户并部署数据库程序
2. 配置 cluster.ini，设置新节点的 NID、ROLE、PORTS 等参数
3. 更新所有现有节点的 cluster.ini，加入新节点信息
4. 启动新节点，集群自动识别并开始数据迁移均衡

**扩容特点**：
- 存储容量或 I/O 能力不足时，按需增加存储节点
- 存储节点增加后，集群数据自动进行数据迁移均衡
- 新增节点可同时作为工作节点分担访问压力
- 性能并发近线性增长

### 与主流数据库扩容对比

| 对比项 | XuguDB | Oracle RAC | MySQL Cluster | PostgreSQL Citus |
|--------|--------|------------|---------------|------------------|
| 扩容方式 | 在线添加节点 | 需 Grid Infrastructure 支持 | 在线添加数据节点 | 添加工作节点 + rebalance |
| 数据均衡 | 自动迁移均衡 | 依赖 ASM rebalance | 自动再分布 | 需手动 rebalance_table_shards |
| 业务影响 | 对业务透明 | 需规划维护窗口 | 数据重分布期间有影响 | rebalance 期间可能影响性能 |

## 故障切换

### 管理节点故障

- 双活冗余设计，两个管理节点同时运行
- 一个管理节点故障时，另一个自动无缝接管
- 无需人工干预

### 存储节点故障

当任意存储节点服务停止：

1. 立即通知管理节点
2. 管理节点选择故障节点上主版本数据对应的任一数据副本，切换为主版本
3. 通知主版本向其余未停止服务的存储节点复制还原丢失的数据版本
4. 继续提供数据访问服务，避免单点故障导致业务中断

### 工作节点故障

- 用户连接可转移到其他工作节点
- 所有工作节点对等，无需访问特定服务端

### 与主流数据库故障切换对比

| 对比项 | XuguDB | Oracle RAC | MySQL Cluster | PostgreSQL |
|--------|--------|------------|---------------|------------|
| 管理节点故障 | 自动接管（双活 M 节点） | CRS 自动恢复 | 管理节点单点（需冗余部署） | 依赖 Patroni/Keepalived |
| 存储节点故障 | 副本自动切换 + 数据重建 | ASM 冗余保护 | NDB 副本自动切换 | 流复制切换（需工具） |
| 计算节点故障 | 连接自动迁移至其他 W 节点 | TAF 透明切换 | SQL 节点重连 | 需应用层处理 |
| RTO | < 30s（同城灾备） | 秒级（TAF） | 秒级 | 取决于 Patroni 配置 |
| RPO | 0（同城方案） | 0（同步模式） | 0（同步副本） | 取决于复制模式 |

## 数据均衡

### 自动均衡机制

XuguDB 采用 8MB 物理 CHUNK 自动切片，数据通过内部算法分片并均衡分布：

- 主副版本依据内部存储落盘规则，打散均衡分布到每一个存储节点
- 新增存储节点后，集群自动执行数据迁移均衡
- 通过 `STORE_WEIGHT` 参数控制各节点的存储权重

### 存储权重配置

在 cluster.ini 中通过 `STORE_WEIGHT` 参数配置：

- 若各服务器磁盘容量不同，按磁盘容量比配置
- 若集群磁盘容量相同，配置为同一数值

### 与主流数据库数据均衡对比

| 对比项 | XuguDB | Oracle RAC | MySQL Cluster | PostgreSQL Citus |
|--------|--------|------------|---------------|------------------|
| 分片方式 | 8MB CHUNK 物理自动切片 | ASM 条带化 | 基于分片键哈希 | 基于分布列哈希 |
| 均衡触发 | 自动（扩容后自动均衡） | ASM rebalance | 自动 | 手动 rebalance |
| 业务维度切割 | 否（物理 CHUNK） | 否（ASM 块） | 是（按分片键） | 是（按分布列） |
| 权重控制 | STORE_WEIGHT 参数 | ASM 磁盘组配置 | 不支持 | 不支持 |
