# 集群管理

## 概述

XuguDB 集群由多个节点组成，每个节点可承担一个或多个角色，通过职责划分与标准通信协议实现高可用、易扩展的分布式架构。

## 一、节点角色

### 1.1 角色说明

| 角色 | 标识 | 职责 |
|------|------|------|
| **控制角色**（Master） | M | 管理集群元信息与运行状态，节点注册、版本管理、故障处理、数据均衡、全局锁服务 |
| **查询角色**（Query） | Q | 建立 TCP 监听端口，对外服务入口；只有含 Q 角色的节点才能接受外部客户端连接，不含 Q 的节点仅参与集群内部通信 |
| **工作角色**（Work） | W | 并发并行任务调度与计算处理，生成执行计划，与存储角色交互完成数据读写 |
| **存储角色**（Storage） | S | 数据持久化存储，响应读写请求 |
| **变更记载角色**（Gather） | G | 异步接收工作角色提交的变更操作，用于审计、备份与增量恢复 |

### 1.2 角色部署约束

| 角色 | 约束 |
|------|------|
| 控制角色 | 仅一组（主+备），必须在两个独立节点，节点号相邻（建议 1、2 号） |
| 查询+工作角色 | 必须部署在同一节点（QW 组合），Q 提供 TCP 监听对外服务，W 提供任务调度计算，缺一不可 |
| 变更记载角色 | 最多部署在两个节点 |

### 1.3 请求执行流程

1. 客户端 → 查询角色（接入、解析、校验）
2. 查询角色 → 工作角色（转发执行）
3. 工作角色 → 控制角色（获取元信息）
4. 工作角色 ↔ 存储角色（数据读写）
5. 工作角色 → 变更记载角色（异步记录变更）

## 二、状态与配置查看

### 2.1 查看集群状态

```sql
-- 方式一：SHOW 命令（无需切换系统库，无需权限）
SHOW CLUSTERS;

-- 方式二：系统虚表（需在系统库，需权限）
SELECT node_id, node_ip, node_type, node_state FROM SYS_CLUSTERS;
```

两者数据一致，均来自当前节点内存中的集群元信息。

### 2.2 配置文件

集群配置保存在 `cluster.ini` 文件中，包含全局配置与各节点的本地配置。

**配置同步机制**：
- 各节点启动后加载本地 `cluster.ini`
- 非主控节点加入集群时从主控节点拉取最新元信息并更新
- 运行中通过心跳检测与广播机制保持一致

## 三、节点管理操作

所有操作须在**系统库**中以 **SYSDBA** 或 **DBA 权限**用户执行。

### 3.1 语法

```sql
-- 添加节点
ALTER CLUSTER ADD NODE node_id DESCRIBE 'node_desc';

-- 下线节点（V12+）
ALTER CLUSTER SET NODE node_id OFFLINE;

-- 删除节点（须先下线）
ALTER CLUSTER DROP NODE node_id;
```

### 3.2 node_desc 格式

用空格分隔的键值对：

```
'RACK=0001 PORTS="192.168.2.236:19114" ROLE=SQW LPU=3 STORE_WEIGHT=3 STATE=DETECT'
```

### 3.3 示例

```sql
-- 添加 4 号节点
ALTER CLUSTER ADD NODE 4 DESCRIBE 'RACK=0001 PORTS="192.168.2.236:19114" ROLE=SQW LPU=3 STORE_WEIGHT=3 STATE=DETECT';

-- 验证
SELECT node_id, node_ip, node_type, node_state FROM SYS_CLUSTERS;

-- 下线并删除
ALTER CLUSTER SET NODE 4 OFFLINE;
ALTER CLUSTER DROP NODE 4;
```

> **注意**：不允许直接删除活跃节点。应先下线，确认所有存活节点日志输出 `Node i deaded done` 后再删除。

## 四、集群扩容

### 4.1 非分布式升级为分布式

适用于单机迁移到分布式集群：

1. 使用分布式版本安装包搭建新集群
2. 使用迁移工具将单机数据导入新集群（离线迁移）
3. 切换应用连接至新集群

### 4.2 离线扩容

适用于分布式系统增加节点（需停服务）：

1. 停止服务，在所有现有节点 `cluster.ini` 中追加新节点配置
2. 将现有节点 BIN 和 SETUP 目录拷贝到新节点，修改端口和节点号
3. 启动所有节点，自动完成注册、同步和版本修复
4. 通过 `SHOW CLUSTERS` 验证

### 4.3 在线扩容

适用于运行中的多节点集群（不中断业务）：

1. 执行 `ALTER CLUSTER ADD NODE ...` 添加节点信息
2. 准备新节点（拷贝 BIN/SETUP，修改配置）
3. 启动新节点，自动注册到集群
4. 通过 `SHOW CLUSTERS` 验证

### 4.4 存储版本修复

扩容后系统自动执行版本修复：

- 新节点加入后，若原有版本不足，自动创建数据版本
- 后台异步执行，不阻塞整体读写

```sql
-- 判断版本修复是否完成（结果为 0 表示完成）
SELECT COUNT(*) FROM SYS_GSTORES WHERE store_sta != 41;
```

## 五、节点降级（防脑裂）

### 5.1 触发条件

同时满足以下条件时控制节点进入降级：
1. 集群注册节点总数 > 1
2. 仅剩本节点存活，其他节点均心跳超时

### 5.2 降级行为

- 进入只读模式，禁止所有写操作（DDL/DML）
- 拒绝用户写请求和节点加入

由参数 `enable_node_degrade` 控制。

### 5.3 降级恢复（三节点集群示例）

1. 定位并解决根因（网络/节点故障）
2. 重启除主控节点外的所有节点
3. 在主控节点清除降级标志：`SET CLUSTER_FAULT_LEVEL TO 0;`
4. 通过 `SHOW CLUSTERS` 验证所有节点状态

## 六、节点故障与恢复

### 6.1 故障排查

| 检查项 | 说明 |
|--------|------|
| 硬件故障 | 磁盘、电源、主板 |
| 系统日志 | dmesg、/var/log/messages |
| 数据库日志 | EVENT.LOG 等 |
| 资源使用 | CPU/内存，关注 OOM |

### 6.2 恢复方式

**方式一：重启故障节点**（机器可访问时）
- 修复问题后重启，系统自动恢复心跳和元信息同步

**方式二：替换为新节点**（原节点无法恢复时）

```sql
-- 1. 删除故障节点
ALTER CLUSTER DROP NODE node_id;

-- 2. 准备新机器，部署一致的软件与配置

-- 3. 添加新节点
ALTER CLUSTER ADD NODE node_id DESCRIBE 'node_desc';

-- 4. 启动新节点，系统自动恢复数据版本
```

## 七、常见问题

| 问题 | 原因 | 解决建议 |
|------|------|----------|
| 节点无法加入集群 | MY_NID 设置错误 / 端口被占用 | 检查配置和端口 |
| 启动后状态 INVALID | 心跳连接未建立 | 检查网络和防火墙 |
| 添加节点命令报错 | 节点号已存在 / 非系统库 / 权限不足 | 检查编号、库和权限 |
| 删除节点命令报错 | 目标节点未下线 / 非系统库 / 权限不足 | 先下线再删除 |
