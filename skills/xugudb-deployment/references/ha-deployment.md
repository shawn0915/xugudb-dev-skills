# 集群高可用部署方案

## 概述

XuguDB 基于 Shared-Nothing 架构，提供四级高可用部署方案，从单机房到两地三中心逐级递进。

## 方案一：数据中心部署（单机房）

### 架构

- 存算分离：计算节点与存储节点独立部署
- 管理节点双活：2 个 Master 节点自动故障切换
- 多副本存储：数据自动分布到多个存储节点

### 特性

| 指标 | 值 |
|------|-----|
| RPO | 0 |
| RTO | 秒级 |
| 最少节点 | 3 |
| 适用场景 | 单机房高可用 |

### 部署要点

- Master 节点必须配置 2 个（主备）
- 存储节点建议 3 个以上（保证 3 副本）
- 计算节点根据负载配置

## 方案二：同城双活

### 架构

- 两个对称数据中心
- 实时双向同步
- 两个中心均可处理读写请求

### 特性

| 指标 | 值 |
|------|-----|
| RPO | 0 |
| RTO | 秒级 |
| 最少节点 | 6（每中心 3） |
| 适用场景 | 跨机房负载均衡 |

### 部署要点

- 两个数据中心对称部署
- 每个中心都有完整的 Master + 工作 + 存储节点
- 数据实时双向同步
- 任一中心故障，另一中心自动接管

## 方案三：同城灾备

### 架构

- 主中心处理所有读写
- 备中心通过 Binlog 增量同步
- 备中心可提供只读查询

### 特性

| 指标 | 值 |
|------|-----|
| RPO | 0 |
| RTO | < 30 秒 |
| 最少节点 | 6 |
| 适用场景 | 机房级容灾 |

### 部署要点

- 主中心正常部署
- 备中心配置 Binlog 同步
- 故障切换时备中心升级为主中心
- 恢复后重新建立同步关系

## 方案四：两地三中心

### 架构

- 同城两个中心（灾备模式）
- 异地一个中心（异步复制）
- 防范区域级灾难

### 特性

| 指标 | 值 |
|------|-----|
| RPO | 近 0（异地有延迟） |
| RTO | 分钟级 |
| 最少节点 | 9 |
| 适用场景 | 区域级容灾 |

### 部署要点

- 同城部分按"同城灾备"方案部署
- 异地中心通过异步复制同步数据
- 异地 RPO 取决于网络延迟和带宽

## 方案选择指南

| 需求 | 推荐方案 |
|------|----------|
| 最基本的高可用 | 数据中心部署（方案一） |
| 跨机房负载均衡 | 同城双活（方案二） |
| 机房级容灾 | 同城灾备（方案三） |
| 最高级别灾难恢复 | 两地三中心（方案四） |
| 成本敏感 | 企业版双节点 |
| 开发测试 | 标准版单机 |

## 副本配置

```sql
-- 查看当前副本数
SHOW default_copy_num;

-- 设置副本数（建议生产环境 >= 3）
SET default_copy_num = 3;

-- 安全副本数（写入时最少同步副本数）
SET safely_copy_num = 2;
```

### 版本副本限制

| 版本 | 最大副本数 |
|------|-----------|
| V11 | 4 |
| V12 | 16 |

## 脑裂防护

- 控制节点与所有存储节点失联时自动进入只读模式（降级）
- 避免双写导致数据不一致
- 网络恢复后需手动清除降级标志：

```sql
SET CLUSTER_FAULT_LEVEL TO 0;
```

## 网络建议

- 节点间建议使用万兆网络
- 建议使用 BOND 双网卡做网络冗余
- 同城双活/灾备的跨机房链路建议 ≤ 1ms 延迟
- 异地复制需考虑带宽和延迟对 RPO 的影响
