# 产品架构

## 软件架构

虚谷数据库采用自研分布式通信协议构建底层架构，核心设计特点：

- **节点对等处理机制**：打破传统主从架构限制，所有节点具备同等的通信与协作能力
- **角色灵活配置**：按需实现单机、双机高可用或分布式集群等多样化部署形态
- **单进程多线程架构**：数据存储、SQL 解析、事务管理等核心功能封装为独立线程组，通过内部通信机制高效协同

### 核心特性

1. 支持事务 ACID 特性，可同时满足 **OLTP** 和 **OLAP** 场景需求，支持 HTAP 混合型业务负载
2. 具备数据强一致性和高可用性保障，所有节点均支持高性能事务处理
3. 支持在线动态扩展与负载均衡
4. 提供完善的安全保障机制
5. 高度兼容 Oracle、MySQL、PostgreSQL 等主流数据库的 SQL 语法

## 无共享架构（Shared-Nothing）

虚谷数据库是**原生分布式架构**数据库，采用 Shared-Nothing（完全无共享）架构进行集群设计：

- 物理资源无共享
- 数据切片存储

### 与主流数据库架构对比

| 特性 | XuguDB | Oracle RAC | MySQL Cluster | PostgreSQL |
|------|--------|-----------|---------------|------------|
| 架构类型 | Shared-Nothing | Shared-Everything | Shared-Nothing | Shared-Nothing (Citus) |
| 原生分布式 | 是 | 否（共享存储） | 是（NDB） | 否（需插件） |
| 存算分离 | 支持 | 不支持 | 不支持 | 不支持 |
| 自动分片 | 物理 CHUNK 自动切片 | 需手动配置 | 需配置分片键 | 需 Citus 配置 |

## 集群角色

集群中按功能职责划分为五类角色：

| 角色 | 职责 | 说明 |
|------|------|------|
| **主控管理角色（M）** | 集群管理、心跳检测、全局锁仲裁（检测死锁并解锁） | 不参与用户请求响应，必须配置 2 个 |
| **查询角色（Q）** | 建立 TCP 监听端口，对外服务入口 | 只有含 Q 的节点才能接受外部客户端连接，**必须与 W 组合部署** |
| **工作计算角色（W）** | 并发并行任务调度与计算处理，生成执行计划 | 负责任务调度和数据读写，**必须与 Q 组合部署** |
| **存储管理角色（S）** | 存储数据库数据，实现数据持久化、存储高可用、数据快速定位 | 数据持久化层 |
| **变更收集角色（G）** | 收集集群数据变更信息，记载信息粒度达到事务级 | 可选部署，最多 2 个节点 |

> **注意**：一台物理服务器可部署一个或多个数据库角色。

## 存算融合部署

虚谷支持在一台物理服务器上同时部署多个数据库角色，让一个节点的数据库进程执行多个角色能力，保证**所有节点都可以提供读写服务**。

## 中心管理-对等处理架构

- 所有 QW 节点均可接收用户请求（Q 角色建立 TCP 监听对外服务，W 角色负责任务调度计算），无需访问特定服务端
- 用户连接任一 QW 节点，均可访问其权限内可访问的所有数据
- 当 QW 节点负载过高时，可通过增加 QW 节点分担负载压力

## 切片化存储管理

数据存储节点**不以逻辑表和业务维度进行数据切割**，而是以切分物理 CHUNK 的方式进行数据存储：

- 数据通过系统内部算法进行分片处理并存储到各个存储节点
- 数据存储以 **8MB** 进行自动切片
- 主副版本依据内部存储落盘规则，将所有存储打散均衡分布到每一个存储节点中

### 故障处理

当任意存储节点服务停止：
1. 立即通知管理节点
2. 管理节点选择故障节点上主版本数据对应的任一数据副本，切换为主版本
3. 通知主版本向其余未停止服务的存储节点复制还原丢失的数据版本
4. 继续提供数据访问服务，避免单点故障导致业务中断

### 在线扩展

存储容量或 I/O 能力不足时：
- 按需增加存储节点
- 存储节点增加后集群数据自动进行数据迁移均衡
- 扩展节点同时作为工作节点分担访问压力

## 变更收集

- 变更收集节点可按需部署
- 提供事务级粒度的变更收集
- 用户可通过系统内置包解析变更信息
- 解析后可提供字段数据输出或 SQL 输出
- 可用于同构或异构表的数据恢复与同步
- 支持按事务提交时间进行日志记录和时间回溯
